## 2. Layout-usage variability (generalizing the v0.5.0 cap)

The post-mortem deck used `executive_summary` 9× in 30 slides. v0.5.0 capped that one layout; this release generalizes the principle with three new S3 gate checks, designed around the HANDOFF rule *"vary across themes, stay consistent within a theme"*:

1. **`layout_share_cap`** — no single layout may exceed **25% of content slides** (floor allowance of 2; decks under 6 content slides exempt; `executive_summary` keeps its stricter 15% check; `two_column_text`/`memo_text` have hard caps instead).
2. **Theme tags** — slides may declare `"theme": "case-studies"` in `content.json`. A themed series counts as **one** occurrence toward the share cap: five case-study slides that look identical are discipline, not monotony. This is the consistency mechanism.
3. **`theme_consistency`** — all slides sharing a theme must use the *same* layout; a theme spanning two layouts fails with a listing of the offenders.

Documented in SKILL.md (Gate S2), `planning-guide.md` §5 (with a JSON example), `known-pitfalls.md` #18–19, and the schema's `global_constraints` block. All enforced mechanically — no verbal rules.

## 3. Audit findings

### 3.1 Speed — verdict: not a bottleneck, no code changes warranted

Measured on this machine (30-slide deck):

| Operation | Time |
|---|---|
| Content gate (subprocess incl. interpreter start) | ~0.4 s |
| Render gate (subprocess, full geometric QA) | ~0.9 s |
| `eng.save()` + `full_cleanup()` XML sanitization | ~60 ms |
| `api-schemas.yaml` YAML parse (dominant gate cost) | ~0.3 s |
| Full test suite (44 tests) | ~21 s |

The real cost drivers per run are **token loading** (below) and **gate round-trips** — the latter is the v0.7.0 render-gate auto-fix task, unchanged.

## 4. English-only hardening (no CJK, no Tencent)

- **Upstream check:** `likaku/Mck-ppt-design-skill` has no releases and only a README edit since our `v2.3.3-harness-v2` fork point. Nothing to port. Its v2.3.0 review/auto-fix pipeline is already in our tree as `review.py`.
- **Mandarin purge:** `review.py` carried the upstream **Chinese-language auto-fix regexes** (hedging removal, sentence compression, CJK punctuation handling) — kept "for a future bilingual fork" but guaranteed no-ops for English decks. Replaced with conservative English equivalents (`"due to the fact that"→"because"`, hedging-opener removal, clause trimming on `;`/`,`) — the auto-fix path now actually does something. The `.skill` zip was rebuilt so the published artifact is clean too.
- **Tencent:** already fully removed in v0.2.0 (Hunyuan cover-image integration and `tencentcloud-sdk-python` deleted). Verified: the engine contains **zero network calls** of any kind. The existing brand-leakage CI scan blocks `tencent`/`hunyuan` strings.
- **New CI guard:** a `CJK scan` step now fails the build if any tracked source file contains literal CJK characters (escape sequences used for *detection* logic don't trip it). The guarantee is mechanical, not a convention.
